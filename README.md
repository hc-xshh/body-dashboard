# Body Dashboard

Daemon 的个人身体数据仪表盘。自动解析体测截图，结合最新体测、当日周计划与历史趋势，生成一个适合每天查看的 **状态 → 执行 → 趋势** 页面。

**正式线上地址**：`https://body-dashboard.pages.dev`

## 当前功能

### 1. 顶部摘要区
- 展示最新体测日期 / 时间
- 展示体型、身体年龄、当天训练主题
- 展示**体测同步状态双态提示**：
  - 已同步：当天建议基于今日最新体测
  - 未同步：当天建议沿用最近一次体测，并明确提示日期

### 2. 阅读路径导航
- 提供可点击的阅读路径：**状态 → 执行 → 趋势**
- 适合每天快速打开后按固定顺序浏览

### 3. 状态区
- 🃏 核心指标卡片：体重、体脂率、BMI、基础代谢、内脏脂肪、水分、骨量、综合得分
- 🕸️ 身体成分雷达图
- 💡 基于 **Phase 3A 已启动** 的规则引擎健康建议（短中长窗口 / 个人基线 / baseline band / 主策略 / 阶段状态 / 证据链 / 训练负荷语义）

### 4. 执行区
- 🏋️ 今日训练执行单
- 🧴 今日护肤流程
- 🥗 今日推荐饮食

### 5. 趋势区
- 📊 体重 & 体脂率趋势折线图
- 💪 肌肉 / 水分 / 去脂体重趋势
- 📋 历史记录表格

## 数据来源与生成逻辑

- 体测数据存储在 `src/data/measurements.json`
- 由 Hermes Agent 解析体测截图后更新数据
- 当日训练 / 护肤 / 饮食内容来自本地计划配置
- 健康建议由规则引擎生成（阈值 + 趋势窗口 + 主策略判断 + 个人基线 + baseline band + 训练负荷 / intake 语义）
- 如果当天没有新体测，页面仍可用，但会明确提示当前建议使用的是最近一次体测数据

## 每天内容是怎么刷新的

这个项目当前不是后端驱动，也不是页面运行时去请求外部 API，而是：

- **静态数据文件**：保存体测历史
- **本地周计划**：保存训练 / 护肤 / 饮食模板
- **前端规则计算**：按今天日期 + 最新体测 + 最近趋势生成当天页面

可以理解成：

> 周计划内容会按日期自动切换；
> 体测数据相关内容会随着最新截图写入而更新。

### 1. 哪些内容是预先存好的

#### 体测历史
- 文件：`src/data/measurements.json`
- 内容：日期、时间、体重、体脂、BMI、肌肉、水分、骨量、得分、身体年龄、体型等
- 用途：
  - 顶部“最新记录”
  - 核心指标
  - 趋势图
  - 历史记录表
  - 饮食微调与健康建议的输入

#### 周计划配置
- 文件：`src/data/dailyPlans.js`
- 预置内容：
  - 每周训练计划（周一到周日）
  - 每周护肤计划（晨间固定 + 夜间按星期切换）
  - 饮食基础模板

#### 健康规则
- 文件：`src/utils/healthAnalysis.js`、`src/utils/rulesEngine.js`
- 预置内容：
  - 体脂 / 内脏脂肪 / 水分 / 基础代谢 / 骨量 / 得分的阈值判断
  - 基于最新记录 + 短中长趋势窗口的信号提取
  - 个人基线（baseline）偏离判断
  - baseline 波动带（baseline band）与带内 / 带外位置判断
  - 主策略模式（稳定执行 / 代谢保护 / 收紧冗余热量 / 恢复优先 / 先观察）
  - 阶段状态（stateStage）
  - 训练负荷语义（trainingLoad）与 intake 策略（intakeStrategy）
  - 分组证据链与置信度输出

### 2. 哪些内容会每天自动变化

#### 今日星期与训练主题
页面会按 `Asia/Shanghai` 时区计算今天是周几，然后自动切换：
- 今天训练主题
- 今日护肤流程
- 阅读路径中的当天说明

也就是说，即使你今天没有发新体测，**训练 / 护肤这类按周计划驱动的内容也会自动更新到今天对应的版本**。

#### 顶部同步状态提示
页面会检查 `measurements.json` 里最新一条记录的 `date` 是否等于今天：
- 等于今天：显示“今日体测已同步”
- 不等于今天：显示“今日未同步，沿用最近一次体测”

所以这里判断的不是云端同步状态，而是：**本地数据文件里是否已经存在今天的体测记录**。

### 3. 今日饮食为什么看起来不是完全固定的

饮食模块不是简单写死一张卡，而是：

- 先从 `dailyPlans.js` 里的基础饮食模板出发
- 再根据以下因素做规则微调：
  - 今天是力量日 / 下肢日 / 有氧日 / 恢复日
  - 最新体脂、内脏脂肪、水分、基础代谢、骨量
  - 最近几次体重 / 体脂 / 肌肉 / 水分 / BMR 的趋势

因此它是：

> **固定模板 + Phase 3A 规则引擎微调**

而不是纯静态，也不是实时联网生成。

### 4. 每天真正的体测更新是怎么发生的

当前项目里，页面本身不会自己去体脂秤 App 拉数据，也没有内置数据库或定时抓取服务。

现在的更新链路是：

1. 你发送新的体测截图
2. Hermes Agent 触发体测同步技能，识别截图中的结构化指标
3. 更新 Obsidian 记录
4. 更新 `src/data/measurements.json`
5. 本地 `npm run build`
6. `git commit` / `git push origin main`
7. Cloudflare Pages 自动构建部署
8. 线上页面刷新为最新内容

所以项目层面并不是“网页自己每天采集新数据”，而是：

> **你发送截图 → Agent 更新数据源 → 前端页面读取新数据并重新部署**

### 5. 如果今天没发体测，会发生什么

页面仍然可正常打开，且会继续自动切到今天对应的训练 / 护肤 / 阅读路径；
只是与身体状态强相关的部分（核心指标、饮食微调、健康建议、趋势）会继续基于最近一次已有体测来展示。

这也是顶部同步状态提示存在的意义：
它会明确告诉你当前建议到底是基于“今天的新数据”还是“最近一次历史数据”。

## 防回归与安全回退

### 空数据不再白屏

本轮已增加一个专门的状态推导层：

- `src/utils/dashboardState.js`
- `tests/dashboard-state.test.js`

页面现在会先判断 `measurements.json` 是否有可用记录：

- 有数据：正常进入 **状态 → 执行 → 趋势** 视图
- 无数据：进入安全回退页，明确提示“暂无体测数据”，不会白屏，也不会生成误导性的训练/饮食建议

### 最小回归测试

```bash
node --test tests/dashboard-state.test.js
node --test tests/rules-engine.test.js
```

### 推荐发布前验证

```bash
node --test tests/dashboard-state.test.js tests/rules-engine.test.js
npm run build
```

如果你修改了体测同步逻辑、`measurements.json` 结构或首页状态推导，至少要重新跑一次上面的命令，确认：

1. 空数据回退仍然成立
2. 规则引擎旧行为没有回归
3. 生产构建仍然通过
